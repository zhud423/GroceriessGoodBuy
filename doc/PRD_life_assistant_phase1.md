# 生活管家 Web 一期 PRD（v0.1）

- Date: 2026-03-23
- Status: Draft
- Product Stage: 自用型 MVP
- Product Form: Web 应用

## 1. 产品定位

生活管家一期是一个面向个人/家庭内部小范围使用的生活效率工具。

当前产品目标不是商业化平台，不是导购工具，也不是完整库存系统，而是围绕线上购物场景，帮助用户低成本完成三件事：

1. 记录买过或想买的商品。
2. 用简单标签表达体验判断。
3. 通过轻量库存状态和购物记录，减少忘买、重复买、找不到历史记录的问题。

## 2. 当前一期要解决的问题

用户当前主要在盒马、小象、七鲜、山姆等平台线上购物，但存在以下问题：

1. 买到过好商品，之后想复购时记不清具体是哪一款。
2. 历史购物截图很多，但商品信息分散，难以结构化沉淀。
3. 同一个商品可能在多个平台重复出现，容易重复记录。
4. 家中库存没有最低限度的状态管理，补货依赖记忆。
5. 缺少“单次购物”这一层记录，无法回溯某次采购买了什么。

## 3. 目标用户

一期只服务于当前产品创建者本人及少量熟人测试用户。

用户特征：

1. 主要在线上平台购买日用品、食品、生鲜或家庭消耗品。
2. 愿意整理生活信息，但无法接受高录入成本。
3. 更关注“我之后还能找到它”和“我知道该不该补货”，而不是复杂的数据分析。

## 4. 一期目标与非目标

### 4.1 一期目标

1. 让用户能从购物截图快速沉淀商品信息。
2. 形成统一的商品主库，避免重复建商品。
3. 让商品可以按分类、平台、标签、库存状态等维度筛选。
4. 支持记录每一次购物，并将购物记录与商品关联。
5. 支持最轻量的评分/判断标签体系。
6. 对同一商品保留多平台最近成交价，并对可称重商品计算标准化单价，支持横向比较。

### 4.2 一期非目标

以下明确不属于当前一期：

1. 自动比价、优惠券计算、自动复购。
2. 在线抓取商品详情页或自动搜索更多商品图。
3. 社区、推荐流、评论区、多人协作。
4. 专业库存管理能力，如批次、保质期、预警算法、采购对账。
5. 复杂的 AI 决策建议或自动替用户下单。

说明：

- “后续自动在线搜索商品图”是未来可扩展能力，不是当前一期 requirement。
- 当前一期只提取截图中已经存在的商品图作为商品首图。

## 5. 核心对象

一期围绕四类核心对象展开：

1. 商品（Product）
2. 购物记录（Shopping Record）
3. 商品分类与标签（Category / Tag）
4. 库存状态（Inventory State）

### 5.1 关于“清单”是否作为独立实体

当前一期不建议把“清单（List）”建成独立业务实体。

原因很直接：

1. 当前产品的核心长期对象是商品。
2. 用户现在说的“清单”，本质上大多是对商品库按规则聚合后的结果。
3. 例如“待补货”“调味品”“山姆买过的”“避雷商品”，都可以通过筛选条件或系统视图生成。

因此，一期建议把清单定义为两类能力：

1. 系统视图，例如待补货、最近购买、回购商品。
2. 保存筛选，例如“调味品 + 山姆 + 回购”。

只有当后续出现明确需求，例如“本周采购单”“露营采购单”“春节囤货单”这种需要人工维护、临时组织、可增删条目的集合时，才有必要引入独立的 `List` 实体。

## 6. 信息架构

建议一期采用以下一级导航：

1. 首页
2. 商品库
3. 购物记录
4. 待补货
5. 截图导入

### 6.1 首页

首页作为总览页，展示：

1. 待补货商品
2. 最近新增商品
3. 最近购物记录
4. 常用筛选入口，例如“调味品”“清洁”“山姆”“避雷”

### 6.2 商品库

商品库是核心主库，不区分“已买”和“未买”的数据结构。

商品库需支持：

1. 列表浏览
2. 搜索
3. 多维筛选
4. 新增/编辑商品
5. 查看商品详情

### 6.3 购物记录

购物记录是“单次购物”的容器对象，用来承载一次订单、一次采购、或一次截图导入事件中的多件商品。

购物记录需支持：

1. 浏览历史记录
2. 查看某次购物包含哪些商品
3. 从购物记录跳转到关联商品
4. 查看来源平台和原始截图

### 6.4 待补货

该页面是商品库的一个系统视图，用于集中展示库存状态为“快没了”或“已用完”的商品。

### 6.5 截图导入

用于上传购物平台截图，完成识别、校对、去重匹配、入库和关联购物记录。

## 7. 商品模型（Product）

### 7.1 商品核心字段

建议一期商品至少包含以下字段：

1. 商品唯一标识 `product_id`
2. 商品名称
3. 标准化名称（系统内部，用于去重）
4. 一级分类
5. 平台集合（盒马 / 小象 / 七鲜 / 山姆，可多选）
6. 商品首图（当前仅取截图中的图片）
7. 规格信息（如 500g / 1L / 2 件装）
8. 标签集合
9. 库存状态
10. 备注
11. 创建时间
12. 最近一次关联购物时间

### 7.2 商品分类

当前已经确认分类只做一级，不做二级。

一期建议直接采用常规线上超市的一级分类口径，并做轻量收敛，避免过细。

建议一期默认分类为：

1. 蔬菜水果
2. 肉禽蛋
3. 海鲜水产
4. 乳品烘焙
5. 熟食速食
6. 粮油调味
7. 酒水饮料
8. 休闲零食
9. 厨房用品
10. 洗护清洁
11. 家居日用
12. 其他

说明：

- 分类是一期检索效率的基线能力，不是可有可无的增强项。
- 一期先使用系统预设一级分类，不开放二级分类。
- `其他` 是必要兜底类，避免识别结果无法归类。

### 7.3 商品标签

一期评分不做星级或分数体系，统一使用文字标签表达。

建议默认内置一组基础标签：

1. 推荐
2. 回购
3. 一般
4. 避雷
5. 性价比高
6. 性价比低

说明：

- 一期标签只使用系统预设，不支持用户自定义。

## 8. 商品唯一标识与去重规则

这是当前一期的核心基础能力，不是未来优化项。

### 8.1 目标

当用户第一次识别某商品后，系统创建一个商品主档。

之后再次从截图中识别到同一商品时：

1. 不重复新建商品。
2. 将本次识别结果关联到已有商品。
3. 将本次导入挂到对应购物记录下。

### 8.2 一期建议规则

一期建议采用“系统内部唯一 ID + 可解释的去重匹配规则”的方案：

1. `product_id` 由系统生成，作为唯一主键。
2. 同一商品跨平台默认合并为同一个商品主档，不按平台拆分商品。
3. 去重匹配优先参考：
   - 标准化商品名
   - 规格
   - 品牌信息（若截图中可识别）
   - 历史平台别名
4. 当匹配置信度高时，自动关联已有商品。
5. 当匹配结果不确定时，进入人工确认，让用户决定“关联已有商品”还是“新建商品”。

说明：

- 商品主档只有一个，但可关联多个平台的成交记录。
- 平台价格不作为商品拆分依据，而是作为购物记录维度的数据保留。

### 8.3 当前不建议做法

当前一期不依赖：

1. 条码作为唯一去重依据
2. 在线商品库同步
3. 外部电商 API

因为这些能力并不是当前 Web MVP 成立的前提条件。

## 9. 购物记录模型（Shopping Record）

### 9.1 定义

购物记录表示一次具体的购物事件。

当前一期中，一次购物记录严格对应一次订单。

一次购物记录下可以包含多个商品，并保留这次购物的来源与上下文。

### 9.2 一期建议字段

1. 购物记录唯一标识 `shopping_record_id`
2. 平台
3. 购物时间
4. 原始截图集合
5. 备注
6. 商品项列表
7. 创建时间

### 9.3 商品项（Record Item）建议字段

1. 关联商品 `product_id`
2. 识别出的商品名称原文
3. 成交价格
4. 数量
5. 规格原文
6. 重量信息（若可识别）
7. 标准化单价（如每 100g 价格，按规则自动计算）
8. 本次是否新建商品
9. 本次是否命中已有商品

说明：

- 商品是长期对象。
- 购物记录是一次性事件对象。
- 一个商品可以出现在多次购物记录中。
- 价格数据保留在购物记录商品项中，不单独建设商品价格历史表。
- 对肉、菜等可称重商品，系统应在识别到价格和重量后自动计算每 100g 价格，便于跨平台比较。

### 9.4 平台价格展示规则

一期不做完整价格历史能力，但商品详情页应能展示该商品在各平台最近一次已知成交价。

建议规则：

1. 价格来源于购物记录商品项。
2. 每个平台只展示最近一次已知成交价作为当前参考。
3. 对可称重商品，同时展示最近一次每 100g 价格。
4. 不提供历史价格曲线或时间趋势分析。

## 10. 库存管理模型（简化版）

一期库存只做轻量状态管理，不做精细盘点。

### 10.1 一期目标

1. 让用户知道哪些商品需要补货。
2. 不增加明显维护负担。

### 10.2 一期建议字段

每个商品有一个库存状态字段，建议初版支持：

1. 充足
2. 快没了
3. 已用完
4. 未知

### 10.3 一期库存更新方式

当前建议以人工更新为主：

1. 用户在商品详情页手动更新库存状态。
2. 用户在购物记录入库后，可选择把对应商品状态更新为“充足”。

说明：

- 自动推断库存消耗不是当前一期 requirement。
- 待补货视图应直接由库存状态驱动。

## 11. 截图识别能力

### 11.1 当前识别范围

一期截图识别的目标不是“完全自动化”，而是“尽量减少手动输入”。

当前一期只支持订单详情页截图导入。

对于订单详情页截图，系统应尽量提取：

1. 平台来源
2. 商品名称
3. 商品图片
4. 规格信息
5. 价格
6. 数量
7. 购物时间

### 11.2 一期保存流程

建议截图导入采用以下流程：

1. 用户上传一张或多张属于同一订单的截图
2. 系统按单次订单合并识别截图内容
3. 系统展示识别结果供用户校对
4. 系统执行商品去重匹配
5. 用户确认每个商品是“关联已有商品”还是“新建商品”
6. 保存购物记录
7. 更新商品主库

### 11.3 一期原则

1. 所有识别字段都必须允许人工修改。
2. 去重不确定时不能静默合并。
3. 截图原图要保留，便于回查。
4. 多张截图必须允许合并到同一条订单记录中。
5. 任何识别失败或缺失的必填信息，都由用户人工补录后再保存。

## 12. 商品库筛选能力

一期商品库至少支持以下筛选维度：

1. 关键词搜索
2. 一级分类
3. 平台
4. 标签
5. 库存状态
6. 是否有购物记录
7. 最近购买时间

说明：

- 筛选不是附属能力，而是商品库成立的核心检索能力。

## 13. 一期关键页面

### 13.1 页面一：首页

目标：帮助用户快速进入最常用动作。

建议模块：

1. 待补货商品
2. 最近购物记录
3. 最近新增商品
4. 快捷导入截图入口
5. 常用筛选标签

### 13.2 页面二：商品库列表页

目标：浏览、筛选、搜索、进入商品详情。

### 13.3 页面三：商品详情页

目标：查看商品完整信息并进行编辑。

建议展示：

1. 商品图
2. 名称与分类
3. 平台
4. 标签
5. 库存状态
6. 各平台最近一次已知成交价
7. 对可称重商品展示最近一次每 100g 价格
8. 最近购买记录
9. 历史购物记录列表
10. 备注

### 13.4 页面四：购物记录列表页

目标：按时间查看历史购物事件。

### 13.5 页面五：购物记录详情页

目标：查看某次购物买了什么，以及每个商品是否已入库或已匹配。

### 13.6 页面六：截图导入页

目标：完成上传、识别、校对和保存。

## 14. 核心用户流程

### 14.1 流程一：首次从截图导入商品

1. 用户上传购物截图
2. 系统识别出多个商品项
3. 用户校对字段
4. 系统尝试匹配商品主库
5. 对未命中的商品创建新商品
6. 生成一条订单级购物记录
7. 将商品项与购物记录关联

### 14.2 流程二：再次识别到已有商品

1. 用户上传新截图
2. 系统识别商品项
3. 系统命中已有商品
4. 用户确认关联
5. 保存购物记录，不重复建商品

### 14.3 流程三：查找某商品是否值得复购

1. 用户在商品库搜索关键词
2. 通过分类、平台、标签筛选范围
3. 进入商品详情页查看标签、备注和购买历史
4. 决定是否复购

### 14.4 流程四：查看是否该补货

1. 用户进入待补货页面
2. 查看库存状态为“快没了”或“已用完”的商品
3. 进入商品详情或加入下一次购物决策

## 15. 一期成功标准

从自用 MVP 角度，一期是否成立主要看以下指标：

1. 用户是否愿意持续通过截图导入，而不是放弃记录。
2. 商品主库是否能明显减少重复记录。
3. 用户是否能在 10 秒内找到“之前买过的某样东西”。
4. 待补货视图是否足以支持日常补货决策。

## 16. 风险与设计原则

### 16.1 主要风险

1. 截图识别准确率不稳定，导致校对成本过高。
2. 商品去重规则不清晰，导致主库脏乱。
3. 分类体系如果过重或过细，会直接增加维护负担。
4. 多图订单如果识别或合并不稳定，会直接影响导入体验。
5. 库存维护动作太多，用户中途放弃。

### 16.2 应对原则

1. 先保证可校对，而不是追求全自动。
2. 先建立稳定去重规则，再追求更智能识别。
3. 先做稳定的一级分类，再视实际使用情况扩展。
4. 先支持同一订单的多图合并，再考虑其他截图类型。
5. 库存管理只保留“对补货有用”的最低限度信息。

## 17. 未来版本方向（非当前一期）

以下只作为未来扩展方向记录，不纳入当前一期排期：

1. 自动在线搜索更多商品图
2. 接入条码识别增强去重
3. 价格趋势与平台比价
4. 自动生成购物建议
5. 多人家庭共享

## 18. 当前结论

当前一期已经确认的关键规则如下：

1. 同一商品跨平台合并为一个商品主档。
2. 商品价格保留在购物记录商品项中，商品详情页仅展示各平台最近一次已知成交价。
3. 肉、菜等可称重商品，在识别到价格和重量后自动计算每 100g 价格。
4. 分类与标签都先使用系统预设。
5. 库存只使用离散状态：充足 / 快没了 / 已用完 / 未知。
6. 截图导入当前只支持订单详情页。
7. 同一订单支持多张截图合并识别。
8. 购物记录严格对应一次订单。
9. 一期不支持愿望清单。
10. 一期不建设商品价格历史能力。
11. 所有未识别出的信息统一由用户人工补录。

## 19. MVP 功能优先级

### 19.1 P0：一期上线必做

以下功能缺少任意一项，一期产品都不算成立：

1. 订单详情页多图上传，并合并为同一订单识别草稿。
2. 从截图中识别平台、商品名称、价格、数量、规格、购物时间。
3. 对识别结果进行人工校对和补录后再保存。
4. 商品去重匹配，并允许用户决定“关联已有商品”或“新建商品”。
5. 建立商品主库，并支持按关键词、分类、平台、标签、库存状态筛选。
6. 商品详情页支持编辑分类、标签、库存状态、备注。
7. 购物记录列表页和详情页，能查看一次订单下的全部商品项和原始截图。
8. 商品详情页展示各平台最近一次已知成交价。
9. 对可称重商品自动计算每 100g 价格。
10. 待补货视图。
11. 支持手动新增商品和手动新增订单。

### 19.2 P1：有余力再做

这些能力对产品有价值，但不应阻塞一期上线：

1. 首页统计卡片，例如本月订单数、最近高频复购商品。
2. 商品库的更多排序方式，例如按最近购买、按购买次数排序。
3. 导入历史页，用于查看失败或未完成的识别任务。
4. 商品合并工具，用于后期整理重复商品。

### 19.3 明确延期到后续版本

1. 愿望清单。
2. 价格趋势图和价格历史分析。
3. 购物车页、商品详情页、订单列表页识别。
4. 在线补图。
5. 自动消耗推断库存。

## 20. 页面级详细需求

### 20.1 首页

首页是进入系统后的总览页，目标是让用户快速继续未完成动作。

P0 页面模块：

1. 快捷入口：`导入订单截图`、`手动新增商品`、`手动新增订单`
2. 待补货商品区：展示库存状态为“快没了”或“已用完”的商品
3. 最近新增商品区
4. 最近订单区

P1 可选模块：

1. 本月订单数
2. 最近 30 天高频购买商品

### 20.2 商品库列表页

页面目标是“找东西”和“进入编辑”。

P0 要求：

1. 默认展示商品卡片或表格列表。
2. 支持关键词搜索，搜索范围至少覆盖商品名称和备注。
3. 支持筛选：分类、平台、标签、库存状态。
4. 支持排序：最近新增、最近购买。
5. 支持进入商品详情页。
6. 支持手动新增商品。

列表项建议展示：

1. 商品图
2. 商品名称
3. 分类
4. 平台标签
5. 标签
6. 库存状态
7. 最近购买时间

### 20.3 商品详情页

页面目标是查看商品完整上下文，并进行维护。

P0 信息模块：

1. 基础信息：名称、分类、平台、规格、备注
2. 标签区域
3. 库存状态区域
4. 各平台最近一次已知成交价
5. 对可称重商品的每 100g 价格
6. 最近订单信息
7. 历史订单列表
8. 原始截图回溯入口

P0 编辑能力：

1. 修改名称
2. 修改分类
3. 修改标签
4. 修改库存状态
5. 修改备注

当前一期不在商品详情页内直接编辑历史订单商品项价格。

### 20.4 购物记录列表页

页面目标是按时间回看订单。

P0 要求：

1. 支持按时间倒序展示订单。
2. 支持按平台筛选。
3. 支持进入订单详情页。
4. 支持手动新增订单。

列表项建议展示：

1. 平台
2. 购物时间
3. 商品数量
4. 订单截图缩略图

### 20.5 购物记录详情页

页面目标是完整回看一次订单。

P0 要求：

1. 展示平台、购物时间、备注。
2. 展示该订单的全部商品项。
3. 每个商品项可跳转到对应商品详情。
4. 展示订单原始截图集合。
5. 若某商品项当时匹配到了已有商品，应能标明。
6. 若某商品项是当次新建商品，应能标明。

商品项建议展示：

1. 商品图
2. 商品名称原文
3. 成交价格
4. 数量
5. 规格原文
6. 每 100g 价格（如适用）

### 20.6 截图导入页

该页面是一期的核心生产页面。

建议采用 4 步结构：

1. 上传截图
2. 识别结果
3. 去重匹配确认
4. 保存完成

P0 要求：

1. 允许上传多张同一订单截图，并支持排序。
2. 识别后展示订单级信息和商品项级信息。
3. 所有字段可人工修改。
4. 去重不确定的商品必须逐个确认。
5. 保存前必须补齐必填字段。
6. 保存后生成订单记录，并同步更新商品库。

### 20.7 手动新增商品页/弹窗

这是截图识别失败时的兜底能力，不属于“可选增强”。

P0 要求：

1. 支持输入商品名称、分类、平台、标签、库存状态、备注。
2. 商品图允许为空。
3. 保存后进入商品详情页。

### 20.8 手动新增订单页/弹窗

P0 要求：

1. 支持填写平台、购物时间、备注。
2. 支持添加多个商品项。
3. 商品项可关联已有商品，也可现场新建商品。
4. 支持上传订单截图作为补充凭证，但截图不是必填。

## 21. 字段与校验规则

### 21.1 商品字段规则

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 商品名称 | 是 | 保存前必须有值 |
| 一级分类 | 是 | 只能从系统预设分类中选择 |
| 平台集合 | 否 | 可由历史订单自动累积 |
| 商品首图 | 否 | 当前优先使用截图中商品图，失败时允许为空 |
| 规格信息 | 否 | 从截图识别或人工补录 |
| 标签集合 | 否 | 只能从系统预设标签中选择 |
| 库存状态 | 是 | 默认值为 `未知` |
| 备注 | 否 | 用户自由填写 |

### 21.2 订单字段规则

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 平台 | 是 | 一期只允许选择系统支持的平台 |
| 购物时间 | 是 | 识别不到时必须人工补录 |
| 原始截图集合 | 否 | 截图导入场景下建议保留；手动建单时可为空 |
| 备注 | 否 | 用户自由填写 |

### 21.3 订单商品项字段规则

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 关联商品 | 是 | 保存前必须关联到已有商品或新建商品 |
| 商品名称原文 | 是 | 用于保留当次识别文本 |
| 成交价格 | 是 | 识别失败时需人工补录 |
| 数量 | 是 | 识别失败时需人工补录；默认值可为 `1` |
| 规格原文 | 否 | 用于保留当次规格信息 |
| 重量信息 | 否 | 可识别则保留 |
| 每 100g 价格 | 否 | 仅在价格和重量可计算时自动生成 |

### 21.4 导入保存前校验

截图导入点击保存前，系统至少需要校验：

1. 订单平台已确认。
2. 订单购物时间已确认。
3. 每个商品项都已完成“关联已有商品”或“新建商品”决策。
4. 每个商品项都存在有效商品名称。
5. 每个商品项都已补齐成交价格和数量。
6. 新建商品项都已补齐商品名称和分类。

若任一条件不满足，禁止保存，并提示缺失项。

## 22. 核心业务规则

### 22.1 商品去重规则

1. 系统先基于标准化名称、规格、品牌、平台别名给出候选商品。
2. 高置信度候选默认选中，但仍允许用户改判。
3. 低置信度候选不自动合并，必须人工选择。
4. 用户一旦确认某次匹配结果，该结果应沉淀为后续去重参考。

### 22.2 商品与平台关系

1. 商品是跨平台统一主档。
2. 平台只是订单和价格的来源维度。
3. 商品页展示“出现过的平台”和“各平台最近一次已知成交价”。

### 22.3 每 100g 价格计算规则

系统仅在可确定总重量时计算标准化单价。

建议公式：

1. `总重量（g） = 可解析重量 × 数量`
2. `每100g价格 = 成交价格 / 总重量 × 100`

适用前提：

1. 价格存在
2. 重量信息存在
3. 重量单位可转换为 g

如果任一条件不满足，则不展示每 100g 价格。

### 22.4 库存状态规则

1. 新建商品默认库存状态为 `未知`。
2. 导入订单后，允许用户把对应商品一键更新为 `充足`。
3. `待补货` 视图仅展示 `快没了` 和 `已用完` 两种状态。

### 22.5 手动补录规则

1. 截图识别出的任何字段都允许修改。
2. 截图未识别出的任何必填字段都必须人工补录。
3. 图片无法可靠提取时，不阻塞保存，但需保留原始截图用于回查。

## 23. 一期验收口径

### 23.1 场景一：首次导入新商品

验收标准：

1. 用户上传同一订单的多张截图。
2. 系统识别出订单和商品项。
3. 用户能补齐缺失字段。
4. 保存后生成 1 条订单、若干条商品项、若干个新商品。

### 23.2 场景二：再次导入已有商品

验收标准：

1. 同一商品再次在别的平台出现时，系统能给出已有商品候选。
2. 用户确认关联后，不重复创建商品。
3. 商品详情页能看到新增的平台最近成交价。

### 23.3 场景三：查看补货

验收标准：

1. 用户把某商品改为 `快没了`。
2. 该商品立即出现在待补货视图中。
3. 用户把状态改回 `充足` 后，待补货视图同步消失。

### 23.4 场景四：比较可称重商品价格

验收标准：

1. 导入肉类或蔬菜订单时，若价格与重量可识别，系统自动算出每 100g 价格。
2. 商品详情页可看到不同平台最近一次每 100g 价格。
