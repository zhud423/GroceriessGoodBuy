# 架构决策记录：导入订单的“草稿持久化”与“前端缓存”模式对比

## 1. 背景
本项目具备“截图识别存入草稿 -> 确认 -> 生成正式订单”的能力。在架构设计上，需要明确为什么选择将“草稿持久化存入数据库”，而不是采用“前端缓存草稿 -> 一次性生成正式订单”的轻量级模式。

## 2. 当前设计方案：草稿持久化
当前流程为：
`截图上传 -> AI识别 -> 创建 ImportSession (DRAFT) -> 生成 ImportItemDraft (持久化到DB) -> 用户审核/修改每个 Draft -> Commit -> 入库成正式 Order + Product`

### 关键特征
- 草稿作为核心实体在数据库中有对应表结构（`ImportSession`, `ImportItemDraft`）。
- 草稿具备独立的 ID，可通过专门的一套 API（CRUD）进行更新和状态流转（例如 `PENDING_REVIEW`, `MATCHED_EXISTING`, `CREATE_NEW` 等）。

## 3. 方案对比

| 维度 | 方案A：草稿持久化（当前设计） | 方案B：前端缓存 → 直接生成正式订单 |
|---|---|---|
| **数据存储** | Draft 写入 DB，有独立表和 ID | 识别结果存前端 state / sessionStorage |
| **断点续传** | ✅ 关掉浏览器或杀掉应用后重新打开可继续编辑 | ❌ 刷新、切后台被杀或关闭页面数据丢失 |
| **多端协同** | ✅ 支持手机拍照上传，电脑端大屏审核修改 | ❌ 仅限当前设备当前会话使用 |
| **审计与追溯** | ✅ 可对比 AI 原始识别结果与用户的最终修改 | ❌ 只有最终确认结果，修改过程不可追溯 |
| **复杂编辑适用性** | ✅ 适合需要与商品库做匹配、增删改商品等重决策流程 | ⚠️ 仅适合简单、无需太多交互的“看一眼就确认”场景 |
| **开发复杂度** | 较高（需管理草稿表结构、状态机、多套 API） | 较低（前端维护状态，只暴露一个最终提交接口）|
| **服务端压力** | 较高（每次编辑操作都需要与后端进行交互写入） | 较低（只在最终提交时发生一次写入） |
| **数据一致性** | 后端强校验、多端状态同步可靠 | 依赖前端校验状态，提交时容易发生并发或数据过期冲突 |

## 4. 为什么本项目更适合方案A（草稿持久化）？

结合本项目的核心业务逻辑：
1. **AI 识别并非 100% 准确**：用户在大部分情况下需要逐项检查 `rawName`，与后端的 `candidateProducts` 列表进行匹配比对，选择现有商品或创建新商品，这是一个**重度决策和操作**的交互流程。
2. **重度依赖服务端状态**：在匹配阶段，草稿数据需要关联已有商品库。如果是纯前端缓存，很难在用户频繁交互、修改时将未保存的临时数据与成千上万条后端商品做精准和高性能的匹配。
3. **多图与大数据量**：一次导入可能包含多张较长购物截图的识别记录。这种量级的数据（包含图片、分析结果、对应的不同商品条目等）存放在前端，无论是内存占用还是可靠性都无法保证。
4. **数据的核心价值**：作为比价/购物类应用，账单和价格记录是非常核心的资产。如果遇到浏览器崩溃或异常退出，数据丢失对用户体验的打击极大；持久化草稿提供了可靠的安全网。

## 5. 方案B（前端缓存）适用于什么场景？
相比之下，如果未来的某些子项目或需求符合以下特征，则更适合采用前端缓存模式：
- **轻量级、秒级确认**（如：扫码加好友、名片图片识别后直接添加到通讯录）。
- **即时化要求高**（如：外卖结账、扫码点单），用户期望当场解决，没有“下次再继续”的诉求。
- **数据容错性高**（如：简单的灵感速记），丢了不心疼。
- **单设备操作闭环**，不需要在其他设备上接着做。

## 6. 结论
当前选择的**草稿持久化**模型对于这种重前端交互、重数据匹配的“AI识别账单”系统是完全合理的正确选择。在未来的产品迭代中，如果希望能优化操作效率，可以考虑叠加“一键自动匹配并确认全部”等前端快捷操作流程，但底层流转依旧应当依托稳定的草稿持久化机制。
