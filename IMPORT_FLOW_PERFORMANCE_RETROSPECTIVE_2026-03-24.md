## 1. 复盘目的

这份文档记录一次典型的产品设计与技术实现取舍复盘。

复盘的重点不是“截图导入要不要草稿”，而是：

1. 为什么最初选择了 `导入会话 / 草稿 -> 人工确认 -> 正式订单` 这条大方向。
2. 为什么这条方向在实现上逐渐变重，最终导致用户实际操作不可用。
3. 从发现问题到当前结论，我们依次考虑过哪些优化方案，为什么有些保留、有些放弃。
4. 当前应该如何判断后续继续优化的方向。

## 2. 最初发现的问题

在 2026-03-24 的本地真实端到端验证里，用户实际等待时间已经超出了可用范围。

一次完整 `login -> create import session -> upload image -> analyze -> get detail -> commit` 的典型耗时大致是：

1. `create import session`: 约 `8.4s`
2. `upload image`: 约 `14.7s`
3. `analyze`: 约 `69.4s`
4. `get detail`: 约 `12.3s`
5. `commit`: 约 `70.0s`

这里最直接的问题不是“AI 识别慢一点”，而是：

1. 用户在点击分析和提交后都被同步阻塞。
2. commit 里混入了过多数据库和存储操作。
3. 导入商品数一旦从 5 条上升到几十条，原方案会进一步恶化。

结论很明确：

- 当时的链路已经不是“有优化空间”，而是当前产品基线不可用。

## 3. 当时为什么会先选草稿方案

最初选择 `导入会话 / 草稿 -> 确认 -> 正式订单`，大方向本身并不错误。

当时合理的产品出发点主要有三条：

1. AI 识别结果在确认前不可信，不应直接污染正式订单、商品主档和统计。
2. 截图上传、后台识别、人工校对、失败重试，本身就需要一个服务端可恢复状态。
3. 未来移动端和 Web 都需要复用同一条导入服务链路，草稿态更适合作为稳定边界。

所以，`先草稿，后提交` 不是这次问题的根因。

真正的问题是后续实现逐渐偏向：

1. 用重事务和逐条写入去追求“正式表绝对正确”。
2. 默认单次导入商品数不会太大。
3. 更重视数据一次性落稳，而不是用户等待时间和交互连续性。

## 4. 当时讨论过的主要替代方案

在确认原链路不可用后，先后考虑过几类方案。

### 4.1 本地缓存校对，确认后直接提交正式订单

这个方案的优点是：

1. 页面感觉更轻。
2. 提交前不用反复写服务端草稿。

但它不适合当前这条链路，原因是：

1. 截图和 AI 识别本来就在服务端，本地缓存并不能消掉核心耗时。
2. 刷新页面、切端、失败重试、分析完成后回来继续看，都需要服务端持久状态。
3. 一旦要支持后台分析，本地缓存只能当临时体验优化，不能替代服务端草稿。

结论：

- 对纯手动小表单可行。
- 对当前“截图上传 + 后台识别 + 校对”的链路，不足以替代服务端草稿。

### 4.2 直接先存正式订单，后台识别后再修改订单

这个方案的优点是：

1. 用户提交动作可以更快。
2. 似乎能把“导入完成”前移。

但问题是：

1. 正式订单、统计、最近购买、商品聚合会先吃到不完整或错误数据。
2. 后台识别反复改正式订单，会让正式记录语义变脏。
3. 如果又想避免脏数据，最后仍然要为正式订单引入 `DRAFT / PENDING` 隔离状态。

结论：

- 这不是彻底绕开草稿，而是把草稿态塞进 `orders` 表里，换个名字继续维护。

### 4.3 保留服务端草稿，但把分析和提交做轻

这是最终保留并持续演进的方向。

核心判断是：

1. 草稿边界仍然有必要。
2. 真正需要优化的是“草稿和提交的重量”，不是取消草稿。
3. 用户能接受 AI 在后台花时间，但不能接受每次点击都同步卡死。

## 5. 实际优化路径

### 5.1 第一轮：先把长请求改成异步返回

第一步优先处理的是用户被同步卡住的问题。

落地思路：

1. `analyze` 只负责启动后台任务并立刻返回。
2. 页面轮询 `import session` 状态，而不是把用户卡在一次长请求里。
3. `commit` 先缩成最小事务，图片复制、alias、聚合刷新放到后台。

这轮之后，用户体感已经从“按钮点下去卡死”变成“后台处理中，页面还能继续走”。

典型结果是：

1. `analyze` 请求返回从约 `69s` 降到约 `5.8s`
2. `commit` 请求返回从约 `70s` 降到约 `14s`

但问题仍然存在：

1. commit 还是太慢。
2. 商品归档仍然绑在提交后链路上。

### 5.2 第二轮：允许先落 raw line items，不等商品归档完成

第二步明确把“订单提交”和“商品归档”拆开。

当时的核心调整是：

1. 允许 `OrderItem.productId` 为空。
2. 用 `resolutionStatus` 标识 `PENDING / MATCHED_EXISTING / CREATED_NEW_PRODUCT`。
3. 用户确认时先落正式订单和 raw order items。
4. 商品匹配、新建商品、alias、聚合刷新全部后台补。

这一步解决的是：

- 不再要求所有商品在提交前都完成“关联已有商品 / 新建商品”的重决策。

但这轮之后的实测说明，commit 仍然在 `7-8s` 左右。

这暴露出一个更深的问题：

- 即使商品归档移到后台，只要点击提交时还在做大量数据库准备，提交动作仍然不会快。

### 5.3 第三轮：把订单准备前移到分析阶段

第三步是本次复盘里真正改变结构的一步。

新的思路是：

1. 在分析完成后，提前构建一个隐藏的 `prepared order draft`。
2. 用户真正点击提交时，不再创建整张订单，而是把这个隐藏草稿发布成正式订单。
3. 所有重准备时间都算在 AI 分析阶段，而不是算在提交动作上。

这轮落地后，交互语义变成：

1. AI 分析阶段更长，但用户知道系统还在准备。
2. 一旦页面显示“可提交”，说明订单其实已经准备好了。
3. `commit` 不再承担几十个商品项的准备工作，只做一次轻发布。

典型热态实测结果：

1. `analyze` 总耗时约 `89.2s`
2. `commit` 请求返回约 `2.2s`
3. 提交后后台归档完成约 `37.5s`

这个结果说明：

- “把慢时间前移到 AI 分析里，把提交做轻”是当前产品更符合用户感知的方向。

## 6. 过程中暴露出的技术误判

这次复盘里，最值得明确写下来的不是具体代码，而是几个当时的误判。

### 6.1 默认按单条心智收敛实现

最典型的误判就是：

- 默认单次导入商品数不会太大，所以逐条写入、逐条归档、逐条刷新聚合“暂时还能接受”。

这在批量导入场景里不是小优化，而是基线错误。

一旦用户一单有几十个商品：

1. 提交耗时会线性放大。
2. 后台任务量会爆炸。
3. 用户会直接把产品判断为不可用。

### 6.2 正确性目标没有分层

当时的实现更偏向：

- 希望用户点击提交时，正式表已经尽可能完整、正确、归档完毕。

这本身不是坏目标，但没有再往下拆层：

1. 哪些正确性必须在提交前保证。
2. 哪些正确性可以提交后后台补。

结果是：

- 把所有正确性都压进了同一次点击动作。

### 6.3 低估了远程数据库和存储的真实成本

本项目运行在远程 Supabase Postgres + Supabase Storage 上。

这意味着：

1. 逐条数据库写入不是“本机小成本循环”，而是远程往返。
2. 图片复制也不是内存移动，而是存储读写。
3. 多个本地 dev 进程和后台任务叠加时，还会进一步放大连接池问题。

所以“代码上看只有几步”的链路，在线上或远程环境里会明显放大。

## 7. 当前结论

当前结论可以明确写成三条。

### 7.1 原始产品大方向没有错

`导入会话 / 草稿 -> 人工确认 -> 正式订单`

这条边界对当前产品仍然成立，因为：

1. AI 识别前置不确定性仍然存在。
2. 后台识别和人工校对仍然需要服务端可恢复状态。
3. 正式订单仍然不应该直接承接未确认的 AI 结果。

所以，不应把这次问题归因为“草稿方案本身错了”。

### 7.2 真正的问题是草稿和提交被做得过重

当前阶段更准确的判断是：

1. 草稿态有必要保留。
2. 但草稿态必须足够轻。
3. commit 必须是一个快速发布动作，而不是重准备动作。

也就是说：

- 需要优化的是实现重量，而不是推翻草稿边界。

### 7.3 优化方向必须跟使用场景绑定

这次复盘最终沉淀出的原则是：

1. 技术上选单条处理还是批量处理，必须先看产品真实工作流。
2. 如果存在明显批量场景，不能默认“先单条跑通，后续再扩”。
3. 对当前截图导入链路，批量商品本来就是现实场景，不是未来幻想。

这已经不是“可选优化建议”，而是后续设计约束。

## 8. 当前仍未解决的问题

虽然 commit 已经显著变轻，但这条链路还有未完成项。

### 8.1 图片上传仍然偏慢

当前上传到 Supabase Storage 的耗时仍然明显。

这部分尚未做：

1. 上传前压缩
2. 分析用衍生图
3. 长图切片

### 8.2 AI 分析总耗时仍然偏长

当前虽然把时间成功前移到了 AI 分析阶段，但 `80-90s` 的分析时长仍然不理想。

后续最值得优先尝试的是：

1. 分析前压缩和切片
2. 更有针对性的识别轮次
3. 继续减少分析后立即触发的额外数据库准备工作

### 8.3 本地开发环境会放大连接池问题

这次本地调试也暴露出：

1. 多个 `next dev` 同时运行会快速耗尽数据库连接。
2. 本地 `DATABASE_URL / DIRECT_URL` 配置方式会直接影响体验。

这属于开发环境问题，不是产品主链路本身，但会影响判断。

### 8.4 当前迭代目标已从 `commit` 转向 `upload + analyze`

在 `commit` 已经被压到可接受范围之后，当前更值得优先优化的目标，已经变成：

1. 用户上传截图后，尽快进入可见、可编辑的草稿状态。
2. `analyze` 只聚焦在“识别并生成草稿”，不要继续混入过多 `commit` 预构建成本。
3. 页面状态要明确区分“正在分析截图”和“正在准备可提交订单”，不能再都混成同一个 `PROCESSING` 体感。

对应的实现方向也应调整为：

1. 上传完成后立即启动后台分析，而不是等详情页再补一次触发。
2. 分析完成后，若还需要准备 `prepared order`，应作为独立后台阶段继续跑。
3. 用户一旦能开始校对草稿，就不应再因为后台准备提交快照而被误判为“分析还没结束”。

## 9. 后续设计约束

从这次案例出发，后续涉及批量工作流的技术方案，应明确遵守以下约束：

1. 先判断产品真实使用形态，再决定单条处理还是批量处理。
2. 若产品方案未明确、但批量场景现实存在，先提问澄清，不默认单条。
3. 提交动作只承载“必须同步完成”的最小集合。
4. 识别、归档、聚合、别名补写等能后台化的操作，优先后台化。
5. 不确定性高的数据先进入草稿或等价隔离态，不直接污染正式数据。

## 10. 一句话结论

这次复盘的最终结论不是“草稿方案错了”，而是：

`原来的产品边界大方向没错，但后续实现没有充分按真实批量场景把草稿和提交做轻，导致本应可恢复、可校对的链路被做成了一个用户等待过长的重流程。`
